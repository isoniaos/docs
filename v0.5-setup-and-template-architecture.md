# IsoniaOS v0.5 - Setup and Template Architecture

Status: Planning  
Date: 2026-04-29  
Scope: Organization setup and template-based topology creation for v0.5 Design Partner Alpha

## 1. Purpose

This document defines the v0.5 setup and template foundation before organization setup flows are implemented in App Core.

v0.5 needs a design partner to create an organization, review a suggested topology, submit the required protocol transactions, and verify the indexed result without editing seed scripts or relying on Isonia-only local knowledge.

The core setup principle is:

```txt
Templates are not governance authority.
Templates generate a reviewable transaction plan.
Only on-chain transactions create authority.
```

Setup drafts, templates, validations, previews, SDK helpers, and App Core UI state are assistance layers. They may explain or prepare governance topology, but they must not define who has authority. The contracts remain the source of authority, and Control Plane remains an indexing and query layer.

## 2. Authority Boundary

The setup architecture must preserve the existing IsoniaOS boundary:

```txt
Browser draft/template
  -> reviewable setup actions
  -> wallet-signed protocol transactions
  -> contract events
  -> Control Plane indexing/projection
  -> REST read models
  -> App Core verification
```

Draft state is useful only before transactions are submitted. Indexed state is useful only after transactions are observed and projected. Neither draft state nor indexed state replaces contract authority.

The setup UI must always distinguish:

- unsubmitted draft configuration;
- actions waiting for wallet signature;
- submitted transactions waiting for confirmations;
- transactions observed by the indexer;
- projected read model state visible through REST;
- stale or failed indexing/projection states.

## 3. SetupDraft

`SetupDraft` is the editable setup model used by App Core before and during setup execution.

It should be defined in `@isonia/types` because it is shared by templates, SDK helpers, App Core, and docs. It must remain a plain data shape with no React, wagmi, viem, NestJS, database, or Control Plane dependency.

Recommended shape:

```ts
interface SetupDraft {
  draftId: string;
  templateId?: string;
  templateVersion?: string;
  chainId: ChainId;
  status: SetupDraftStatus;
  organization?: DraftOrganization;
  actions: SetupAction[];
  warnings: SetupValidationWarning[];
  createdAt: string;
  updatedAt: string;
}
```

`SetupDraftStatus` should be explicit enough for App Core state transitions:

```txt
editing
ready_for_review
executing
partially_indexed
completed
blocked
abandoned
```

Draft rules:

- a draft may be created manually or from a template;
- every generated action must be visible and editable before execution;
- action ordering must be deterministic;
- temporary client IDs may be used to link draft actions before real on-chain IDs exist;
- wallet addresses, proposal type scopes, timelocks, and metadata URIs must be shown before signing;
- a draft may be stored locally in the browser, but local storage is not authority;
- a draft may be exported/imported later, but export/import is not required for the first v0.5 implementation.

`SetupDraft` should not contain private keys, wallet session data, internal RPC secrets, database identifiers, or hidden template-only permissions.

## 4. TemplateDescriptor

`TemplateDescriptor` describes a reusable setup blueprint. It should also live in `@isonia/types` because templates and generated setup actions cross repository boundaries.

Recommended shape:

```ts
interface TemplateDescriptor {
  templateId: string;
  version: string;
  name: string;
  summary: string;
  description?: string;
  supportedChains?: ChainId[];
  inputs: TemplateInputDescriptor[];
  defaults: TemplateDefaults;
  actionFactoryId: string;
  warnings: SetupValidationWarning[];
}
```

Template rules:

- templates generate `SetupAction[]`;
- templates do not create authority until actions are executed as protocol transactions;
- templates must not hide bodies, roles, mandates, or policy rules;
- template outputs must be editable before any transaction is sent;
- template defaults should be conservative and understandable;
- template labels and descriptions may become metadata references or fallback labels;
- templates must not be represented as a new on-chain concept for v0.5;
- templates must not imply that Isonia recommends a single governance constitution.

Template inputs should be minimal for the first implementation:

- organization name;
- optional organization metadata URI;
- organization admin address;
- initial council holder addresses;
- treasury holder addresses;
- security/veto holder addresses;
- executor holder address or body assignment;
- proposal type routes to enable;
- timelock values.

## 5. SetupAction Types

`SetupAction` is a reviewable unit of intended on-chain setup work. Each action maps to one wallet-signed protocol transaction or to one deterministic transaction preparation step.

Actions should use stable draft IDs before execution and record observed on-chain IDs after indexing.

Shared base fields:

```ts
interface SetupActionBase {
  actionId: string;
  kind: SetupActionKind;
  label: string;
  description?: string;
  dependsOn: string[];
  executionStatus: SetupActionExecutionStatus;
  txHash?: TransactionHash;
  indexedAt?: string;
  warnings: SetupValidationWarning[];
}
```

The base shape above is illustrative. In `@isonia/types`, `txHash` should use the shared transaction hash type if one exists; otherwise define a dedicated string/hex type for transaction hashes.

Required action kinds for v0.5:

```txt
create_organization
create_body
create_role
assign_mandate
set_policy_rule
```

Execution statuses:

```txt
draft
ready
waiting_for_signature
submitted
confirmed
indexed
failed
skipped
```

### 5.1 Create Organization

Creates the organization root in the shared protocol deployment.

Fields:

- organization draft ID;
- admin address;
- metadata URI, if provided;
- fallback name for App Core display;
- expected chain ID;
- expected contract address.

Expected result after indexing:

- `orgId`;
- admin address;
- organization status;
- created block;
- created transaction hash.

All subsequent actions that require `orgId` must depend on this action.

### 5.2 Create Body

Creates a governance body within an organization.

Fields:

- body draft ID;
- parent organization draft ID or indexed `orgId`;
- body kind;
- fallback name;
- metadata URI, if provided;
- active state.

Expected result after indexing:

- `bodyId`;
- `orgId`;
- body kind;
- active state.

### 5.3 Create Role

Creates a role within a body.

Fields:

- role draft ID;
- body draft ID or indexed `bodyId`;
- role type;
- fallback name;
- metadata URI, if provided;
- active state.

Expected result after indexing:

- `roleId`;
- `bodyId`;
- `orgId`;
- canonical `RoleType`.

Role types must use shared `@isonia/types` enums. The canonical emergency role is `EmergencyOperator`; do not introduce an `EmergencyPauser` alias.

### 5.4 Assign Mandate

Assigns a holder to a role with explicit scope.

Fields:

- mandate draft ID;
- role draft ID or indexed `roleId`;
- holder address;
- start time;
- end time;
- proposal type mask or explicit proposal type scope;
- spending limit where relevant.

Expected result after indexing:

- `mandateId`;
- holder address;
- role/body/org linkage;
- active/revoked state;
- proposal type scope.

Mandate assignment must be explicit about time bounds and proposal type scope because route validation depends on eligible holders, not only bodies and roles.

### 5.5 Set Policy Rule

Creates or updates the active policy rule for a proposal type.

Fields:

- organization draft ID or indexed `orgId`;
- proposal type;
- required approval body draft IDs or indexed `bodyId`s;
- veto body draft IDs or indexed `bodyId`s;
- executor body draft ID or indexed `bodyId`, if execution is expected;
- timelock seconds;
- enabled state.

Expected result after indexing:

- policy version;
- proposal type;
- required approval bodies;
- veto bodies;
- executor body;
- timelock seconds;
- enabled state.

Policy rules must preserve v0.1 policy-version semantics. New proposals snapshot the active policy version at proposal creation. Existing proposals must continue to use their stored `policyVersion`.

## 6. Simple DAO+ Template

Simple DAO+ is the first supported v0.5 template.

It is intended for a small design partner team that wants:

- broad governance approval through a general council;
- treasury-specific review for treasury proposals;
- security veto coverage;
- explicit executor authority for supported alpha execution paths.

Default bodies:

| Body | Kind | Purpose |
| --- | --- | --- |
| General Council | `general_council` | General organization approval and standard proposal review |
| Treasury Committee | `treasury_committee` | Treasury proposal approval and treasury-specific review |
| Security Council | `security_council` | Veto coverage and emergency authority |

Default roles:

| Body | Roles |
| --- | --- |
| General Council | `OrgAdmin`, `Proposer`, `Approver` |
| Treasury Committee | `Approver`, `Executor` |
| Security Council | `Vetoer`, `EmergencyOperator` |

Default policy routes:

| Proposal type | Required approvals | Veto bodies | Executor body | Timelock |
| --- | --- | --- | --- | --- |
| `standard` | General Council | Security Council | General Council or Treasury Committee, configurable | Short alpha default |
| `treasury` | General Council, Treasury Committee | Security Council | Treasury Committee | Longer alpha default |
| `upgrade` | General Council | Security Council | General Council, configurable | Longer alpha default |
| `emergency` | Security Council | Security Council | Security Council | Short or zero alpha default, explicitly reviewed |

Simple DAO+ must require the user to review and confirm:

- actual holder addresses for each role;
- whether a separate executor holder is present;
- whether emergency routes are enabled;
- whether zero or short timelocks are intentional;
- whether every enabled policy route has at least one eligible holder for approval, veto, and execution where expected.

The exact numeric timelock defaults should be defined in implementation planning with the design partner. The template architecture should support changing these defaults without changing the authority model.

## 7. Transaction Execution Model

Setup execution is step-by-step. App Core should never batch hidden authority changes behind a single opaque action in v0.5.

Execution sequence:

1. Build or edit `SetupDraft`.
2. Validate the draft and show warnings.
3. Lock the draft for review.
4. Prepare the next executable `SetupAction`.
5. Ask the connected wallet to sign and submit the transaction.
6. Record the transaction hash in local execution state.
7. Wait for chain confirmation according to runtime config.
8. Wait for Control Plane indexing and projection.
9. Resolve on-chain IDs from REST read models.
10. Mark the action `indexed`.
11. Recompute dependent actions with real IDs.
12. Continue to the next action.

Action dependencies must be explicit. For example:

- bodies depend on organization creation;
- roles depend on body creation;
- mandates depend on role creation;
- policy rules depend on the bodies they reference;
- policy route validation depends on mandates and policy rules.

If an action fails:

- App Core must show the failed action and transaction state;
- dependent actions must remain blocked;
- the user may retry when safe;
- retry behavior must not create duplicate draft assumptions;
- idempotency must be handled by protocol behavior and indexed verification, not by pretending the draft succeeded.

## 8. Draft State vs On-Chain and Indexed State

Draft state:

- exists in App Core before or during setup;
- may use temporary IDs;
- may be edited;
- may contain warnings;
- may be stale relative to on-chain state;
- has no governance authority.

On-chain state:

- is created only by successful protocol transactions;
- contains real organization, body, role, mandate, policy, and proposal lifecycle state;
- is the source of governance authority;
- may be ahead of Control Plane while indexing catches up.

Indexed state:

- is derived from contract events;
- powers REST responses, graph views, route previews, and setup verification;
- may be delayed, stale, or failed;
- must be diagnosable through `/v1/diagnostics`;
- must not override contract authority.

App Core should show all three layers when they differ. A body that exists in a draft but is not yet indexed must not be displayed as if it exists on-chain.

## 9. Waiting for Indexing

After each submitted setup transaction, App Core must wait for indexing before moving to dependent actions.

Recommended wait model:

1. Wait for wallet transaction submission.
2. Wait for chain receipt and configured confirmation depth.
3. Poll `/v1/diagnostics` to confirm the relevant contract cursor has reached the transaction block or latest safe block.
4. Poll the specific read model endpoint that should contain the result.
5. Match the result by stable identifiers where available:
   - transaction hash;
   - created block;
   - admin/holder address;
   - metadata URI;
   - proposal type;
   - body/role relationship.
6. Mark the action as indexed only after the expected read model row is visible.

The UI should classify wait states:

- transaction pending in wallet;
- transaction submitted;
- waiting for confirmations;
- waiting for indexer;
- waiting for projection;
- indexed;
- indexing delayed;
- projection failed;
- wrong chain or wrong contract configuration.

The diagnostics surface should help explain stale state, but the final verification for an action should use the relevant read model, not diagnostics alone.

## 10. Validation Warnings

Validation warnings are advisory. They help users avoid broken setup, but they do not define authority.

Warnings should use shared codes in `@isonia/types` so App Core, SDK helpers, tests, and docs can remain aligned.

Required warning codes:

| Code | Meaning |
| --- | --- |
| `missing_approver_mandate` | A required approval body has no active holder with an approver mandate for the proposal type. |
| `missing_veto_mandate` | A veto body has no active holder with a veto mandate for the proposal type. |
| `missing_executor_mandate` | The executor body has no active holder with an executor mandate for the proposal type. |
| `policy_route_without_eligible_holder` | A policy route references bodies or roles that currently have no eligible holder for one or more required responsibilities. |
| `invalid_timelock` | The timelock is negative, unsupported by the protocol, too short for the selected route, or unexpectedly long for alpha operation. |
| `empty_required_approvals` | An enabled policy route has no required approval bodies. |
| `proposal_type_scope_mismatch` | A holder mandate exists but its proposal type scope does not cover the policy route's proposal type. |

Warning shape:

```ts
interface SetupValidationWarning {
  code: SetupValidationWarningCode;
  severity: "info" | "warning" | "error";
  message: string;
  actionId?: string;
  relatedBodyId?: string;
  relatedRoleId?: string;
  relatedMandateId?: string;
  proposalType?: ProposalType;
}
```

Validation should run:

- when a template generates a draft;
- whenever the draft is edited;
- before the draft is locked for review;
- before each action is submitted;
- after each action is indexed, because real IDs and current indexed state may change warnings.

Errors may block execution where the transaction would be structurally invalid. Warnings should allow execution only when the user can intentionally accept the risk and the protocol transaction is valid.

## 11. Affected Repositories

| Repository | v0.5 setup role |
| --- | --- |
| `types` | Own `SetupDraft`, `TemplateDescriptor`, `SetupAction`, setup execution status, template IDs, and validation warning codes. Keep the package dependency-light. |
| `sdk` | Expose typed helpers for setup draft generation, setup validation, REST polling, and viem transaction preparation where useful. Do not become an authority layer. |
| `app-core` | Implement the setup wizard, template selection, action review, wallet transaction flow, indexing wait states, warnings, and verification UI. |
| `docs` | Document setup architecture, design partner setup guide, troubleshooting, known limitations, and template behavior. |
| `control-plane` | Add no setup write APIs. Add read-only REST endpoints only if existing read models cannot verify setup actions or display policy rules. |

Control Plane may need a read-only policy list endpoint such as:

```txt
GET /v1/orgs/:orgId/policies
```

This endpoint is only needed if the existing overview, graph, and route endpoints cannot reliably show configured policy rules during setup verification. If added, it must return shared DTOs from `@isonia/types`, remain REST-only, and reflect indexed on-chain state.

No v0.5 setup flow should introduce backend-defined governance authority.

## 12. Acceptance Criteria

Architecture acceptance:

- setup and template types are defined in `@isonia/types` before cross-repository use;
- templates are documented as transaction-plan generators, not authority;
- every setup action maps to explicit protocol transactions;
- draft state, submitted transaction state, and indexed state are distinguishable;
- validation warning codes are shared and documented;
- Control Plane remains REST-only and read-only for setup support.

Simple DAO+ acceptance:

- the template generates organization, body, role, mandate, and policy rule actions;
- generated actions are visible before submission;
- generated holder addresses and proposal type scopes are editable;
- default routes cover standard, treasury, upgrade, and emergency proposal types where enabled;
- missing approver, vetoer, executor, and proposal scope issues are shown before execution.

Execution acceptance:

- App Core can submit setup actions one by one;
- dependent actions wait for indexed IDs from previous actions;
- App Core waits for indexing after each transaction before proceeding;
- stale indexer/projection states are visible through diagnostics and setup UI;
- failed actions do not mark dependent actions as complete.

Documentation acceptance:

- design partner docs can explain how to review a generated topology before signing transactions;
- troubleshooting docs can distinguish wallet, chain, indexer, projection, and validation problems;
- out-of-scope limits are explicit before external alpha use.

## 13. Out of Scope

The setup/template foundation does not include:

- GraphQL;
- backend setup write APIs;
- backend-created governance authority;
- storing setup drafts as authoritative server state;
- template-specific on-chain authority;
- token voting;
- quadratic voting;
- reputation scoring;
- anti-Sybil or personhood systems;
- arbitrary production execution;
- Safe, Tally, Snapshot, Hats, or external governance integrations;
- billing;
- SaaS tenant administration;
- public signup;
- metadata publishing service;
- theme marketplace;
- plugin marketplace;
- AI copilot;
- production multi-chain support;
- automatic legal/compliance policy generation;
- claiming that Simple DAO+ is a recommended constitution.

Future versions may add richer templates, saved collaborative drafts, batch transaction UX, or partner-specific setup packages. Those additions must preserve the same authority boundary: templates prepare reviewable transactions, and only contract transactions create governance authority.

## 14. Reviewer Amendments Before Implementation

This section was added after architecture review to reduce ambiguity before implementation starts.

### 14.1 v0.5 Supported Environments

v0.5 should support two environments:

1. local development environment;
2. one shared public EVM testnet environment.

v0.5 must not target mainnet or production multi-chain operation.

The exact public testnet can be selected separately. All runtime configuration, diagnostics, and setup docs should make the active chain explicit.

### 14.2 Policy List Endpoint Is Required for v0.5

For setup verification and policy route builder UX, App Core needs a direct indexed policy list.

Add this read-only REST endpoint to Control Plane if it does not already exist:

```txt
GET /v1/orgs/:orgId/policies
```

Requirements:

- return indexed on-chain policy rules;
- include proposal type, policy version, approval bodies, veto bodies, executor body, timelock, enabled state;
- use shared DTOs from `@isonia/types`;
- expose through `@isonia/sdk`;
- remain REST-only;
- never become a backend authority layer.

This endpoint is required for v0.5 because overview, graph, and proposal-route endpoints are not enough for policy setup review.

### 14.3 Organization Admin vs OrgAdmin Role

The protocol-level organization admin address and any `OrgAdmin` role shown in templates must not be confused.

Rules:

- the organization admin address remains the protocol root authority where the contracts define it as such;
- a template-created `OrgAdmin` role must not imply that it changes the protocol admin address unless the contract explicitly supports that behavior;
- App Core must explain which wallet is expected to submit setup transactions;
- setup actions that require organization admin authority should clearly show the required signer.

If this distinction creates confusion in Simple DAO+, prefer `BodyAdmin`, `Proposer`, and `Approver` roles in the General Council and keep the organization admin as a separate setup concept.

### 14.4 Wallet Authority Checks

Before submitting setup transactions, App Core should show a signer warning when the connected wallet is unlikely to be authorized.

Examples:

- creating bodies, roles, mandates, and policies normally requires the organization admin or an equivalent authorized role;
- assigning mandates should show the expected authority;
- setting policy rules should show the expected authority.

Frontend checks are only hints. Contracts remain the final authority.

### 14.5 Setup Action ID Resolution

Setup actions that depend on earlier actions must not guess final on-chain IDs.

Resolution order:

1. submit transaction;
2. wait for receipt;
3. wait for Control Plane indexing;
4. resolve indexed entity by transaction hash where available;
5. otherwise resolve using deterministic context such as organization, relationship, metadata URI, and created block;
6. store resolved real IDs in the draft execution state;
7. unblock dependent actions only after real IDs are resolved.

If a real ID cannot be resolved, dependent actions must remain blocked.

### 14.6 Setup Action Persistence

For first v0.5 implementation, setup drafts may be stored in browser-local state or local storage.

Rules:

- local draft persistence is convenience only;
- it must not be treated as authority;
- it must not contain private keys, secrets, wallet sessions, or RPC credentials;
- if the page reloads, the user should be able to recover or restart the draft safely;
- failed or partially completed drafts must clearly show which actions were already indexed.

Server-side collaborative draft storage is out of scope for v0.5.

### 14.7 First Implementation Order

Implement v0.5 setup in this order:

1. add setup/template shared types in `@isonia/types`;
2. add policy list DTO and Control Plane endpoint if missing;
3. add SDK methods for policies and setup-related reads;
4. add App Core setup route skeletons;
5. implement Simple DAO+ template generation as draft actions;
6. implement setup draft preview and validation warnings;
7. implement step-by-step transaction execution for one action type at a time;
8. add indexing wait and real ID resolution;
9. add policy route builder after basic body/role/mandate setup works;
10. update docs and troubleshooting after each public interface change.

### 14.8 First Codex Implementation Tasks

Recommended first implementation tasks:

1. `types`: add setup/template DTOs, setup action kinds, setup statuses, validation warning codes, and policy list DTO if missing.
2. `control-plane`: add `GET /v1/orgs/:orgId/policies` if missing.
3. `sdk`: add typed `client.policies.list(orgId)` and any setup helper exports that do not require React.
4. `app-core`: add setup wizard route skeleton and template-selection page without write execution.
5. `app-core`: implement Simple DAO+ draft generation and read-only draft preview.
6. `app-core`: implement validation warnings before adding transaction execution.
7. `app-core`: implement step-by-step setup transactions only after draft preview and warnings are stable.

Do not implement the full setup wizard in one large change.
