# Roles And Permissions

Roles tell users who is responsible. Permissions tell users what each role can do.

## Common Roles

| Role | What the role does |
| --- | --- |
| Admin | Configures organization settings and core roles. |
| Reviewer | Reviews proposals before a decision. |
| Approver | Can approve a proposal under the configured route. |
| Executor | Performs or confirms approved actions. |
| Contributor | Submits proposals or completes assigned work. |
| Observer | Reads records without changing them. |

## Reading A Permission

A permission should be easy to read:

```text
Reviewers can review grant proposals.
Approvers can approve grant proposals after review.
Executors can attach proof after an approved action.
```

If a permission is hard to explain, the organization should simplify it or add clearer guidance before using it with real participants.

## Good Practice

- Give people only the permissions they need.
- Use group roles when a responsibility belongs to a council or working group.
- Keep execution permissions narrow.
- Make manual status changes visible.
- Review roles when a template changes.

## Example

Community Grants DAO gives contributors permission to submit proposals. Reviewers can request changes. Approvers can approve or reject. Executors attach proof after approved actions. Observers can inspect the final records.
