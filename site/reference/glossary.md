# Glossary

| Term | Meaning |
| --- | --- |
| Accountability | The practice of tracking who is responsible for follow-through, what status the work has, what evidence exists, and what remains unresolved. |
| Accountability record | A record that connects a decision to a responsible person or group, due date, status, evidence, notes, and review outcome. |
| Activation | The point where an organization has the required setup to use its configured governance process. In contract-backed flows, activation is tied to modeled state. |
| App Core | The IsoniaOS governance console used to view organizations, proposals, evidence, accountability, diagnostics, and configured wallet interactions. |
| Authority | The source that can decide or prove a specific fact. A contract, an external tool, a manual note, and a UI view can have different authority. |
| Authority boundary | The point where one source's power stops. For example, a UI can show a transaction, but it does not override contract state. |
| Community Grants DAO | The example organization used across this guide. |
| Control Plane | The indexing, read-model, diagnostics, and REST API layer. It explains state, but it is not governance authority by itself. |
| Decision | The result of a proposal or approval process, such as approved, rejected, cancelled, expired, or executed. |
| Evidence | A transaction, external record, document link, manual note, milestone proof, or other artifact that supports a claim about what happened. |
| Execution | The action expected after a decision, such as sending a configured transaction, publishing proof, completing a milestone, or recording that no action is needed. |
| Governance lifecycle | The flow from proposal and review through decision, execution, accountability, and memory. |
| Governance memory | The durable record of decisions, context, evidence, outcomes, unresolved questions, and lessons for future participants. |
| Organization | A DAO, working group, grants program, association, or other group using IsoniaOS to structure governance records. |
| Permission | A rule that says who can perform an action, such as creating a proposal, reviewing it, approving it, executing it, or updating evidence. |
| Proposal | A request for the organization to make a decision. A proposal should explain the reason, expected action, review path, and follow-up owner. |
| Read model | A projected API or database view derived from raw events or other sources. It can help users read state but may lag the source it represents. |
| Role | A named responsibility in an organization, such as admin, reviewer, executor, contributor, or observer. |
| Source disclosure | Details that show where a claim came from, how it was checked, and whether it is current, stale, failed, missing, or unknown. |
| Template | A reusable pattern for organization setup, proposal routing, review requirements, roles, permissions, and accountability fields. |
| Verification | The process of checking whether evidence supports a specific claim. |
