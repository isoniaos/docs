# IsoniaOS v0.1 — Technical Scope

## 1. Goal

Create a **private prototype** governance architecture layer for an EVM + TypeScript stack.

v0.1 must prove that:

- a shared protocol deployment can serve many organizations;
- the governance structure can be formalized through `Organization`, `Body`, `Role`, `Mandate`, `PolicyRule`, and `Proposal`;
- the control plane reliably indexes chain events;
- the frontend can show the power map and decision route;
- the architecture is ready for self-hosted and SaaS modes.

## 2. Key decisions

- one deployment → many organizations
- `orgId` as hard isolation boundary
- REST as public API for v0.1
- event-first ingestion
- public app-core
- private SaaS overlay
- replaceable default theme
- minimal npm dependencies

## 3. Included in v0.1

### On-chain

- `GovCore`
- `GovProposals`
- `DemoTarget`
- organizations
- bodies
- roles
- mandates
- policy rules
- proposal lifecycle

### Off-chain

- blockchain ingestion
- raw event storage
- projections
- REST API
- route explanation
- governance graph
- idempotent processing
- replay support

### Frontend

- org list
- governance overview
- proposals list/details
- route explanation UI
- governance map
- theme separation

## 4. Not included in v0.1

- token voting
- anti-Sybil
- reputation
- marketplace
- enterprise SSO
- billing core
- AI copilot
- arbitrary execution engine
- proxy upgradeability

## 5. Acceptance criteria

### Protocol

- at least 2 organizations can be created
- organizations have isolated bodies, roles, mandates, and proposals
- at least 3 proposal types exist
- the proposal lifecycle works end to end
- execution changes the DemoTarget state

### Control Plane

- events are ingested idempotently
- projections can be rebuilt
- the API returns route explanations and the governance graph
- reprocessing an event does not corrupt state
